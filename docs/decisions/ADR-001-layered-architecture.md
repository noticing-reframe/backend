# ADR-001: 레이어드 아키텍처 채택

## 상태
승인됨 (Accepted)

## 컨텍스트
Reframe 백엔드 프로젝트의 아키텍처를 결정해야 합니다. 주요 고려사항:
- AI 서비스(Claude API)와의 통합
- 캐릭터 데이터 관리
- 프롬프트 템플릿 시스템
- 향후 확장성 (DB 연동, 캐싱 등)

## 결정
**레이어드 아키텍처 (Layered Architecture)** 를 채택합니다.

```
Application Layer (유스케이스, API 엔드포인트)
       ↓
Domain Layer (비즈니스 로직, 엔티티)
       ↓
Infrastructure Layer (외부 서비스, DB, API)
```

## 대안 검토

### 1. 단순 MVC
- 장점: 빠른 개발
- 단점: 비즈니스 로직과 인프라 결합

### 2. 클린 아키텍처
- 장점: 완전한 관심사 분리
- 단점: MVP 단계에서 과도한 복잡성

### 3. 레이어드 아키텍처 (선택)
- 장점: 적절한 분리, 이해하기 쉬움, NestJS와 자연스러운 통합
- 단점: 레이어 간 경계가 모호해질 수 있음

## 결과

### 긍정적
- Claude API 연동 코드가 Infrastructure에 격리됨
- 프롬프트 템플릿 시스템을 독립 모듈로 관리
- 테스트 용이성 확보

### 부정적
- 레이어 간 데이터 변환 필요
- 초기 보일러플레이트 증가

## AI 협업 노트
이 아키텍처 결정은 Claude Code와의 대화를 통해 도출되었습니다:
- AI가 여러 아키텍처 패턴의 장단점을 비교 분석
- MVP 단계에 적합한 복잡도 수준 제안
- NestJS 모듈 구조와의 자연스러운 매핑 확인
