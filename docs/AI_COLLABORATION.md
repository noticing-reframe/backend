# AI 협업 개발 리포트

> Reframe 프로젝트의 AI 활용 전략 및 협업 과정 문서

## 개요

Reframe은 **AI-First 개발 방법론**을 적용하여 개발되었습니다. 프로젝트의 기획, 설계, 구현, 테스트 전 과정에서 AI 도구를 적극 활용하여 개발 생산성을 극대화하고, 서비스 자체도 AI 기반으로 동작합니다.

---

## 1. AI 도구 활용 현황

### 1.1 개발 보조 도구

| 도구 | 역할 | 활용 방식 |
|------|------|----------|
| **Claude Code (CLI)** | 메인 개발 파트너 | 코드 생성, 리팩토링, 아키텍처 설계, 버그 수정, 배포 자동화 |
| **Cursor** | AI 에디터 | 실시간 코드 제안, 인라인 수정 |
| **Claude API** | 서비스 핵심 | 캐릭터 매칭, 대화 생성, 1인칭 채팅 |

### 1.2 사용 모델

| 모델 | 용도 | 선택 이유 |
|------|------|----------|
| `claude-3-haiku` | 대화 생성 (프로덕션) | 빠른 응답 속도, 비용 효율성, 충분한 품질 |
| `claude-opus-4` | 개발 협업 | 복잡한 아키텍처 설계, 프롬프트 최적화 |

## 2. AI 협업 프로세스

### 2.1 개발 워크플로우

```
1. 요구사항 정의
   └─ AI와 대화하며 기능 명세 구체화

2. 아키텍처 설계
   └─ AI가 제안한 레이어드 아키텍처 채택
   └─ Infrastructure → Domain → Application 구조

3. 프롬프트 엔지니어링
   └─ 캐릭터 페르소나 설계
   └─ 1인칭 화법 가드레일 구축
   └─ Tool Use 기반 구조화된 출력

4. 구현
   └─ AI가 생성한 코드를 리뷰 후 적용
   └─ 실시간 피드백 반영

5. 테스트 & 배포
   └─ AI 지원 Docker 빌드
   └─ k8s 배포 자동화
```

### 2.2 AI와의 협업 예시

**사례 1: 캐릭터 매칭 알고리즘 설계**

```
[개발자] 사용자 고민에 맞는 캐릭터를 추천하는 로직이 필요해
[AI] TF-IDF 기반 유사도 매칭과 LLM 기반 추론 중 어떤 방식을 선호하시나요?
[개발자] LLM 기반으로 가자, 더 맥락을 이해할 수 있을 것 같아
[AI] Tool Use를 활용해 구조화된 출력을 받는 방식을 제안합니다...
```

**사례 2: 프롬프트 최적화**

```
[개발자] 캐릭터가 자꾸 조언을 해버려
[AI] 시스템 프롬프트에 "조언하지 않는다" 가드레일을 추가하고,
     "결론을 말하지 마라, 에피소드만 공유하라" 규칙을 명시하겠습니다
```

## 3. 프롬프트 엔지니어링 전략

### 3.1 시스템 프롬프트 설계 원칙

1. **페르소나 고정**
   - 캐릭터 이름, 배경, 말투를 시스템 프롬프트 상단에 명시
   - "You ARE {character_name}" 형태로 정체성 고정

2. **가드레일 계층화**
   ```
   [SECTION 1] CHARACTER VOICE & IDENTITY
   [SECTION 2] GROUND EVERY RESPONSE IN YOUR STORY
   [SECTION 3] HOW TO RESPOND
   [SECTION 4] RESPONSE FORMAT
   [SECTION 5] WHAT TO AVOID
   ```

3. **예시 기반 학습**
   - `dialogue_example` 필드로 말투 예시 제공
   - Few-shot 형태로 원하는 출력 스타일 유도

### 3.2 토큰 최적화 전략

| 전략 | 효과 |
|------|------|
| XML 태그 구조화 | 파싱 정확도 ↑, 토큰 효율성 ↑ |
| 대화 히스토리 직접 전달 | 요약 불필요, 맥락 보존 |
| 캐릭터 컨텍스트 1회 주입 | 시스템 프롬프트에만 포함 |
| 병렬 LLM 호출 | 응답 시간 최적화 (4-5개 동시) |

### 3.3 Tool Use 활용

```yaml
# match_characters 도구 정의
name: match_characters
description: Match characters to user's worry
input_schema:
  matched:
    - reason: "30자 이내 한국어"
    - score: 0-100
    - index: 캐릭터 인덱스
```

**장점:**
- 구조화된 JSON 출력 보장
- Hallucination 감소
- 파싱 오류 최소화

## 5. 학습 및 개선 사항

### 5.1 AI 협업에서 배운 점

1. **명확한 요구사항이 핵심**
   - AI에게 모호한 지시는 모호한 결과를 낳음
   - 구체적인 예시와 제약조건이 품질을 결정

2. **반복적 개선이 효과적**
   - 한 번에 완벽한 결과를 기대하지 않기
   - 작은 단위로 피드백하며 점진적 개선

3. **AI의 제안을 비판적으로 검토**
   - AI도 틀릴 수 있음
   - 코드 리뷰와 테스트는 필수

### 5.2 향후 개선 계획

- [ ] 대화 히스토리 요약/압축 (장기 대화 지원)
- [ ] 캐릭터별 Fine-tuning 검토
- [ ] A/B 테스트 기반 프롬프트 최적화
- [ ] 사용자 피드백 기반 품질 개선

## 6. 결론

Reframe 프로젝트는 AI를 **도구**가 아닌 **협업 파트너**로 활용한 사례입니다. 개발 전 과정에서 AI와 지속적으로 대화하며 설계를 발전시켰고, 서비스 자체도 AI를 핵심 기술로 사용합니다.

이 문서는 AI 협업 개발의 실제 사례로서, 향후 유사 프로젝트의 참고 자료가 될 수 있습니다.
